# AI Failure-Mode Review Checklist

Generated by `init-ai-repo` (spec §4.B). Run this checklist when reviewing any
PR that contains AI-generated or AI-assisted code. It targets the failure modes
that are common to model-authored code and easy to miss in ordinary review. It
is offline and deterministic: this is a generated review convention, not a live
CI gate. Pair it with the test/eval gate (`modules/evals.md`) and the
observability audit (`.ai/observability/audit-checklist.md`).

## Hallucinated dependencies

Models invent packages, modules, APIs, and config keys that look plausible but
do not exist.

- [ ] Every imported package, module, and symbol resolves to a real, installed
      dependency (no invented names).
- [ ] Each newly added dependency exists in the registry and is pinned in the
      lockfile/manifest, not just referenced in code.
- [ ] Every called API, method, flag, and config key exists in the version that
      is actually depended on (no hallucinated signatures).

## Slopsquatting

Hallucinated package names get registered by attackers, so a hallucinated import
can resolve to a malicious look-alike (typosquat/slopsquat).

- [ ] Each new dependency name is verified against the official package source —
      not a near-miss or typo'd look-alike of a known package.
- [ ] New dependencies are checked for publisher, age, download count, and
      maintenance signals; suspiciously new or low-trust packages are rejected.
- [ ] No dependency was added solely because the model named it; provenance is
      confirmed independently.

## Inadequate error handling

Model-authored code tends to favor the happy path and swallow or ignore errors.

- [ ] Failure paths are handled, not silently swallowed (no empty catch blocks,
      ignored returns, or bare `except: pass`).
- [ ] External calls (I/O, network, parsing) validate inputs and handle timeouts,
      partial failures, and malformed data.
- [ ] Errors are surfaced with actionable context and the right severity; the
      code fails closed rather than continuing in an undefined state.

## Looks-right / subtle correctness gaps

Code that reads cleanly can still be subtly wrong: off-by-one, boundary, sign,
unit, and edge-case errors that pass a casual read.

- [ ] Boundary and edge cases (empty, single, max, negative, null) are covered
      by a test, not assumed correct because the code "looks right".
- [ ] Logic was verified against the requirement/spec, not accepted on plausible
      appearance; assumptions the model made are stated and checked.
- [ ] Concurrency, ordering, and state-mutation assumptions are validated; the
      change has an eval or test that would fail if the behavior regressed.
